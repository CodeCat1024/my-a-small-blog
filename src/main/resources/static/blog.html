聊聊依赖引入

一、依赖引入

在使用别人的框架时，我们通常都需要将别人的框架引入到自己的开发环境中，比如使用semantic ui时，就需要将它的css源码和js源码引入到我们的文件中，那么，你知道在你引入依赖的时候，原理是什么吗？

二、原理

想象你在做前端页面的开发时，经常要为文字加一些样式，比如颜色，大小，粗细，间距等，你每次都要写下面这么一段代码

.m-text-color {
color: skyblue;
}
.m-text-thin {
font-weight: 300 !important;
}
.m-text-spaced {
letter-spacing: 1px !important;
}
.m-text-lined {
line-height: 1.8;
}

当每次你设置的样式都一样时，你会想到把这份代码拷贝下来，放到某个文件夹，以后要用的话直接拷贝一下就行了

或者，也可以不拷贝，我们可以直接将这份代码保存成my-text-style.css文件，以后要用时，直接在head部分引入一下(假设你将它放在该路径下)

接着，你还可以把更多的样式整合进来，也就是先写好，以后要用的时候，直接在head上引入一下，然后在我们要修改的文本上面加上这个类名就行了 甚至，那些你平时经常要写的导航栏、按钮、目录等，也可以写好放在这个css文件里面，到时候我们直接用就行了，就不需要每次都从头开始写了， 当你集成的样式越来越多，这个就是你自己做的框架了

那现在我们知道了该怎么使用自己的框架了，那别人的框架呢？

其实也是一样的道理，比如说semantic ui框架，我们直接也是直接将它的css文件引入进来我们的开发环境就行了

比如我们可以来看semantic ui的官网上的引用介绍

但是，如果我们直接把这份代码拷贝到自己的文件中，则会出现找不到资源的错误，这是为什么呢？

其实就是因为你没有把代码下载到你本地里，这就像你上面引入./static/styles/my-text-style.css文件，但是这份文件已经被你删除了一样

那么，我们只要把semantic ui官方他的“my-text-style.css”文件下载到本地然后引入就行了，跟我们使用自己的css文件没有任何区别

三、依赖的依赖

那为什么引入了semantic ui之后还需要引入jQuery依赖呢？

其实很好理解，比方说你先使用你的my-text-style框架写了另一份my-slider-style框架，那你以后引入my-slider-style之前就肯定需要先引入my-text-style啦

四、为什么有时候不需要下载源码？

  src="https://code.jquery.com/jquery-3.1.1.min.js"

比如semantic引入jQuery时，为啥引入的是网址而不是文件呢？并且还不需要将jQuery的源码下载到本地，这是为什么呢？而且，打开这个网址，发现里面还是一大堆完全看不懂的东西......

其实，这也跟我们将源码下载到本地然后去引用一样的，只是我们将这些源码放到线上而已，这样就不用一直占据我们本地的空间了

“那为什么这么乱呢？”

因为这是压缩版，网址上的min就代表这是压缩过后的源代码，这样的话文件更小，加载更快

不信的话你可以看看去掉这个网址里面的.min，看看没有压缩过的版本

而且，我们还可以发现，所谓的压缩，不过就是去掉一些空格和换行而已

五、我非要下载到本地怎么办？

其实，明白了原理之后，我们可以很快地猜出，只要我们把网址上的这个源代码拷贝一下，放到我们本地，然后我们再通过平常我们引用自己css文件的方式去引用就可以了，说到底，跟我们使用自己的css和js文件没有任何区别

并且，反过来看，我们也知道了如何将我们自己写的css文件通过这种方式放到线上去

六、总结

无论是哪种引入方式，本质都是一样的，无非是一个在线上，一个在本地，并且我们可以猜到，如果放在线上，没有网络的话，肯定是没有办法加载成功的

但很多时候也没关系，因为用户要访问你的网站本身就需要网络......

因此，我们只要清楚它的原理，就不会被各种形式所迷惑了

无非就是找到源码，引入到开发环境中

是通过绝对路径引入还是相对路径引入，或者源码放在哪里，都是无关紧要的事情
